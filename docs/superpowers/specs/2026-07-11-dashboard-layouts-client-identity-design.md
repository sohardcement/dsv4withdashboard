# DS4 看板布局与调用服务识别设计

## 目标

把现有仅通过色彩变量区分的三种看板主题改为三种独立的信息架构，同时让调用详情能够显示请求来自哪个客户端服务，而不是只有直连 IP。

## 主题

三个主题使用同一份 `/ds4/status` 状态数据、相同的 Context/KV 管理操作及相同的无障碍语义。浏览器保存选择，初始默认值为 `paper`。

### paper：纸面运行报告（默认）

编辑式单栏报告。首屏先给出模型、当前运行结论、上下文与缓存摘要；资源、调用和操作配置以分节说明和少量关键数字跟随。其目标是让用户快速理解“当前是否健康、发生了什么”。

### terminal：深色控制台

紧凑的等宽监控台。顶行是连接与健康状态，随后是运行指标网格和追加式调用事件流；资源、上下文和缓存均以密集数值呈现。其目标是盯盘和排障。

### calm：从容解释型

状态摘要与时间线。先呈现易读的健康结论、上下文余量和资源解释，然后展示调用事件与按服务的汇总。其目标是日常管理和非技术化阅读。

主题之间必须有不同的 DOM 布局、区块排序和信息密度，不能只改变 CSS 色彩变量。所有可见产品文案保持中文；技术单位、API 名称和客户端原始标识可保留原样。窄屏不应产生水平页面溢出。

## 调用服务识别

每一条调用记录新增 `client`（服务/客户端）字段，并在状态 JSON 的 records、按调用方聚合和看板表格中呈现。

识别优先级：

1. 请求头 `X-DS4-Client`，用于明确的服务名，例如 `hanako-agent`、`hermes-agent`、`openclaw`。
2. 标准请求头 `User-Agent`，在没有显式标识时作为兼容回退。
3. `未标识服务`，两个头均不存在或为空时。

服务名仅是客户端自报的观察信息，不是认证身份；仍保留直接 TCP peer IP，不使用 `X-Forwarded-For`。读取值会去除控制字符、限制长度，并用现有 JSON/DOM 转义路径输出。服务端不记录请求正文、提示词或认证头。

调用历史继续是进程内、最多 200 条的窗口；窗口满且全部活动时新请求不进入记录，已有的严格容量策略不变。按服务汇总和看板筛选基于实际记录窗口。

## 测试与文档

- C 服务测试覆盖服务名优先级、长度/控制字符清理、JSON 输出及历史聚合。
- 浏览器测试覆盖默认 paper、三主题有不同布局结构、主题持久化、服务列/筛选与恶意文本不作为 HTML 解析。
- README 记录 `X-DS4-Client` 用法、`User-Agent` 回退、它们不是可信认证身份以及三主题的定位。
