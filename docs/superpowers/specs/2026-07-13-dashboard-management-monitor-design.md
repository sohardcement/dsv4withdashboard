# DS4 Dashboard 管理与监控体验优化设计

## 背景与目标

当前 dashboard 已能展示运行、缓存、上下文、主机和调用信息，并提供三种布局主题。然而默认页面较长，三种布局增加了视觉与交互维护面，配置操作的视觉层级也未充分体现 DS4 作为本地推理系统软件的专业感。

本次改造的优先级是：

1. 配置管理；
2. 实时盯盘与排障；
3. 快速运行概览。

主要痛点是视觉品质，改造同时解决与视觉层级直接相关的信息组织和交互反馈问题。最终界面采用“精密仪表”方向：编辑式排版、工业仪表感、克制配色、清晰数字层级，弱化常见的卡片堆叠。

## 设计决策

dashboard 从三套主题布局收敛为同一视觉系统下的两个任务模式：

- **管理**：默认模式，用于判断当前状态、调整 KV 与上下文配置并确认生效结果。
- **监控**：紧凑模式，用于查看实时性能、调用流、请求详情和主机资源。

两种模式共享同一份 `/ds4/status` 快照、同一套状态格式化函数和同一份管理表单逻辑。模式是信息架构，不是颜色主题；不再保留“纸面运行报告 / 深色控制台 / 从容解释型”三套 DOM。

模式选择保存到浏览器 `localStorage`，键名为 `ds4-dashboard-mode`，合法值仅为 `management` 和 `monitor`。缺失或无效值回退到 `management`。

## 视觉系统

界面保留浅色、偏暖的纸面底色，但从“报告页面”调整为精密系统控制台：

- 深墨色用于主文字和主要操作，氧化红用于活动、风险和关键强调；成功状态使用独立绿色并辅以文字。
- 标题使用系统可用衬线字体栈，数据和标识使用系统等宽字体栈，正文使用系统无衬线字体栈；不加载网络字体。
- 通过字号、字重、标尺线、数字对齐和留白建立层级，减少阴影、圆角胶囊和同质化卡片。
- 数字使用 tabular figures，关键指标保持稳定宽度，避免轮询更新造成明显跳动。
- 状态不能只依赖颜色；必须同时显示文字或图标语义。
- 动效仅用于短暂状态过渡，并服从 `prefers-reduced-motion`。

## 共享框架

顶部栏在两种模式中保持一致，包含：

- DS4 标识；
- “管理 / 监控”模式切换；
- 本机或离线状态；
- 当前阶段或健康状态；
- 最近成功刷新时间。

模式切换使用原生按钮，并用 `aria-pressed` 表示当前模式。切换只改变两个模式根节点的 `hidden` 与 `aria-hidden`，不会重新请求数据或重置筛选、输入和操作状态。

页面级状态分为：

- **在线**：正常轮询并展示最近更新时间；
- **过期**：轮询失败但存在最后快照，保留数据并明确标注“数据已过期”及快照时间；
- **不可用**：首次请求即失败，展示无法读取状态，不把缺失数据渲染为零。

## 管理模式

### 信息架构

管理模式按以下顺序组织：

1. 运行与容量摘要；
2. 磁盘 KV 容量配置；
3. 上下文窗口配置；
4. 最近调用摘要；
5. 主机状态摘要。

桌面宽度下使用窄侧栏锚点导航，快速定位配置和观察区块。窄屏隐藏侧栏，所有内容按同一语义顺序排列。

### 运行与容量摘要

首屏只呈现完成管理任务所需的结论：

- 运行状态、当前阶段和队列；
- 上下文已用、上限、余量和利用率；
- 磁盘 KV 已用、上限和利用率。

完整性能细节不在管理首屏重复，由监控模式承接。

### 配置区块

每个配置区块必须在同一视觉范围内闭环呈现：

- 当前运行值；
- 当前使用量或利用率；
- 可编辑目标值；
- “立即影响运行”或“重启后生效”的明确标签；
- 风险或语义说明；
- 操作按钮；
- 进行中、成功或失败结果。

磁盘 KV 保留两个不同操作：

- **检查并立即应用**：预检后可能改变当前运行时容量；
- **仅保存为下次启动上限**：只写持久化设置，不改变当前进程。

上下文窗口只有“保存下次启动设置”。成功反馈必须同时说明“已保存，需要重启”和“当前运行值未改变”。

### 最近调用和主机摘要

管理模式只显示少量最近调用，字段为请求 ID、服务、API、结果和可用时的耗时。完整筛选、调用方、错误和请求详情位于监控模式。

主机摘要显示内存压力、物理内存和 DS4 RSS；采样不可用时显示“不可用”，不能显示为零。

## KV 管理状态机

KV 立即应用使用显式页面内状态机：

```text
idle → checking → review → applying → success
                       ↘ error
```

### idle

用户编辑目标容量。目标不同于当前值时标注“尚未应用”。输入无效时在字段附近显示具体约束，且不调用服务端。

### checking

调用现有 `dry-run` 接口。KV 输入、单位和两个 KV 操作按钮全部禁用，防止重复操作或“应用 + 保存”交叉执行。反馈显示“正在检查缓存清理压力”。

### review

预检结果在配置区块内结构化展示，包括：

- 旧上限和新上限；
- 修改前使用量；
- 是否需要清理；
- 预计清理条目数和释放空间；
- 明确的“确认立即应用”和“取消”。

不再使用原生 `window.confirm`。风险审阅出现后焦点移入审阅区；取消后焦点回到“检查并立即应用”。

### applying

使用预检返回的修订版本调用现有 `apply` 接口。提交期间继续锁定 KV 控件。

服务端返回 `kv_state_changed` 时最多自动重新预检一次：

- 如果新的清理影响与用户已审阅内容相同，可继续提交；
- 如果容量、预计清理条目或释放空间发生变化，返回 `review` 并要求重新确认；
- 第二次修订冲突进入 `error`，提示等待活动缓存操作结束后重试。

### success 和 error

成功结果保留在配置区块内，展示容量、条目和清理量的前后变化，并等待下一次状态快照确认当前值。结果不会使用自动消失的 toast。

失败时保留用户目标值和可重试路径。403 会将写操作切换为只读并说明仅限本机；离线会禁用写操作，但保留最后快照和未提交输入。

“仅保存为下次启动上限”与立即应用共用事务锁，但不经过清理审阅。结果明确区分已提交、已确认持久化和未确认持久化。

## 监控模式

### 实时指标

顶部指标带只显示服务端现有快照能够证明的即时信息：

- 当前阶段、活动请求和服务；
- 解码速度与进度；
- 预填充速度与进度；
- 当前请求 KV 命中率；
- 上下文利用率；
- 队列与客户端数量。

本次不引入趋势图。服务端没有提供时间序列持久化，dashboard 不应把短期客户端采样伪装成可靠历史。

### 调用流与请求检查器

调用记录是监控模式主工作区。保留按服务、调用方、API 和结果筛选；服务端数据继续限制为最多 200 条。

每行显示请求 ID、服务 / 调用方、API、结果，以及能从 `started_at` 与 `finished_at` 或活动请求快照可靠计算的耗时。行可用鼠标或键盘选择，选中请求在相邻检查器显示现有记录字段：

- 服务与直接调用方；
- API、种类、流式和工具标志；
- token、KV 命中和缓存来源；
- 结束原因和最近错误。

选中请求从当前筛选结果中消失时，选择清空并显示“请选择一条调用记录”，不自动跳到另一条记录。

### 主机资源

主机资源位于监控工作区下方，显示物理内存、可用内存、内存压力、Swap 和 DS4 RSS。不可用采样保持明确的不可用状态。

## 数据流与更新规则

dashboard 继续每秒轮询 `/ds4/status`，并保留当前的 generation 守卫，确保同一时间最多一个状态请求在生效。每个成功快照更新：

- 共享顶部状态；
- 管理摘要和只读当前值；
- 监控指标、调用筛选选项和主机资源；
- 当前选中请求的检查器。

轮询不得：

- 覆盖当前获得焦点的输入；
- 清除未提交的目标值；
- 覆盖 `checking`、`review` 或 `applying` 的操作反馈；
- 自动清除成功或失败结果；
- 解析任何服务端字符串为 HTML。

所有动态文本通过 `textContent` 写入。服务、调用方、错误和其他不可信字符串不得进入 `innerHTML`。

## 响应式与无障碍

- 桌面以 1,240–1,320px 内容宽度为主要设计尺寸。
- 平板下配置区块由双列变为单列，监控检查器移到调用流下方。
- 390px 手机宽度不产生页面级横向溢出；宽表格如需横向空间，只能在有明确边界的表格容器内滚动。
- 交互目标最小高度 44px；所有输入有可见标签和错误关联。
- 模式切换、筛选、调用选择、风险审阅和管理操作均可通过键盘完成。
- 使用清晰可见的 `:focus-visible`，不能用颜色作为唯一焦点或状态提示。
- 操作结果使用 `role="status"` 或适当的 `aria-live`；风险审阅使用可聚焦标题和说明，不伪装为阻塞式系统对话框。
- DOM 标题顺序在两个模式和各响应式断点下保持合理。

## 实现边界

- 保持 dashboard 为 `ds4_server.c` 内的内嵌 HTML/CSS/JavaScript，不引入前端框架、资源服务器、网络字体或新的构建步骤。
- 复用 `/ds4/status`、`POST /ds4/admin/kv-cache` 和 `POST /ds4/admin/context`。
- 不改变管理端点的 loopback、same-origin、`X-DS4-Admin: 1` 和无 CORS 约束。
- 不修改推理、KV 存储、会话或后端执行路径。
- 不加入长期历史、告警系统、远程管理或新的身份认证能力。

## 测试与验收

浏览器测试扩展 `tests/dashboard_ui_test.js`，至少覆盖：

1. 默认进入管理模式，无效模式回退，模式选择持久化；
2. 管理与监控根节点切换、共享状态同步和窄屏无页面溢出；
3. KV 输入校验、预检状态、结构化影响审阅、取消和确认；
4. KV 操作串行化、一次修订冲突重试、影响变化重新确认、二次冲突错误；
5. KV 成功、持久化未确认、畸形响应、403 和离线反馈；
6. 上下文保存成功、失败、403，以及“当前运行值未改变”的文案；
7. 状态轮询不重叠，并且不覆盖焦点输入或活动操作结果；
8. 调用筛选、键盘选择、检查器字段和选择从筛选结果消失时清空；
9. 恶意服务、调用方和错误文本不生成元素或脚本；
10. 1440px、平板和 390px 下的布局与交互可用性。

服务端回归继续使用 `make ds4_test && ./ds4_test --server`，确认 dashboard 路由、状态 JSON 和管理端点不回退。浏览器回归使用 `./tests/run_dashboard_ui_test.sh`。最终生成桌面管理页、桌面监控页和 390px 管理页截图，检查信息层级、截断、焦点、控件状态和溢出。

视觉验收标准：

- 首屏能明确区分当前状态、当前值、目标值和生效时机；
- 配置操作在同一区块内完成，不依赖原生确认框或短暂 toast；
- 监控模式在 1440px 下一屏展示关键指标、至少五条调用记录和选中请求详情；
- 中文产品文案完整，技术单位、API 名称和原始客户端标识可保留原样；
- 颜色、字体、间距、线条和控件遵循“精密仪表”视觉系统。
